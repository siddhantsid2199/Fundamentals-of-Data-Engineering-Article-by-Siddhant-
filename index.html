Fundamentals of Data Engineering

A Comprehensive Guide to Modern Data Architecture, Workflows, and Technologies

Introduction to Data Engineering

Data Engineering serves as the foundational discipline that bridges the gap between raw data collection and actionable business intelligence. While data science focuses on extracting insights and patterns from processed data, data engineering concentrates on building robust, scalable infrastructure that makes analytics possible in the first place.

In today's data-driven landscape, organizations face exponentially growing volumes of information arriving at unprecedented velocities from diverse sources. Data engineers design and maintain the critical systems that ingest, process, store, and serve this data, ensuring seamless flow from operational systems to analytical platforms. Their work enables data scientists, analysts, and business users to access clean, reliable data for decision-making.

The modern data engineering paradigm emphasizes automation, scalability, and reliability. Engineers build pipelines that can handle batch and streaming data, implement data quality controls, and ensure that data governance requirements are met throughout the entire data lifecycle.

Why Data Engineering is Essential

Data Engineering provides structured, relational data architecture that transforms raw, unstructured information into refined, accessible datasets. This structured approach enables organizations to derive meaningful insights and make data-driven decisions effectively.

OLTP vs OLAP Systems

Understanding the fundamental difference: OLTP systems excel at processing high-volume transactional workloads with strong consistency requirements, while OLAP systems are optimized for complex analytical queries across large datasets with flexible schema designs.

Characteristic OLTP (Online Transactional Processing) OLAP (Online Analytical Processing)
Primary Purpose Handle day-to-day transactions and maintain operational data integrity Support business intelligence and analytical queries for decision-making
Data Organization Row-based storage (CSV, AVRO) optimized for frequent updates and inserts Column-based storage optimized for aggregations and analytical operations
Query Patterns Short-duration, high-frequency transactions with predictable access patterns Complex analytical queries involving joins and aggregations across large datasets
Schema Design Highly normalized (3NF) structures to eliminate redundancy and ensure consistency Denormalized designs (Star/Snowflake schemas) prioritizing query performance
Use Cases E-commerce transactions, financial systems, inventory management, CRM systems Business intelligence, data mining, trend analysis, predictive modeling
Data Scope Current operational data focused on recent transactions and active records Historical data spanning multiple time periods for trend and comparative analysis

Modern Data Pipeline Architecture Flow

Data Warehouse Architecture Layers

The concept happens in layers. The first one is staging layer where the data will be extracted from OLTP source and put into staging layer in raw form. Now there are two options in staging as well:

Three-layer architecture: Source extraction → Staging (with storage options) → Core transformation

Staging Layer: Raw Data Landing Zone

Transient Storage (Used 99% time)

Temporary store: The first layer is Transient layer meaning it will temporary store the data and will truncate tomorrow as I am not saving this data. This approach minimizes storage costs and is used in most implementations.

Persistent Storage

Preserve Data & Store History: The other layer is Persistent where I will preserve the data and keep updating the history. This approach enables data reprocessing and audit trails.

Core Integration Layer: Transformation Hub

Then comes the second layer which is called core integration layer where we will transform the data and dimensional data model will come into picture.

Incremental Loading (Data Fetching Strategy)

Incremental loading is a data fetching strategy that processes only new or changed data rather than reprocessing the entire dataset. This approach significantly reduces processing time, storage requirements, and system resources while maintaining data accuracy and completeness.

Incremental Loading Example

Consider a sales data table with daily quantity information. Instead of reloading all historical data every time new data arrives, incremental loading processes only the new records.

Dimensional Modeling

Dimensional modeling is a data design technique used to optimize databases for analytical queries and reporting. This approach organizes data into facts (measurable business events) and dimensions (descriptive context), creating intuitive structures that mirror how business users think about their data.

Core Concepts

Fact Tables

What are Fact Tables? Fact tables contain quantitative business metrics and measurements. They store the "numbers" of business processes - the measurable events that occur in business operations.

Characteristics:
  • Quantitative Data: Sales amounts, quantities, durations, counts, ratios
  • Foreign Keys: References to related dimension tables
  • Additive Measures: Metrics that can be summed across dimensions
  • Large Volume: Typically contain millions or billions of records

Dimension Tables

What are Dimension Tables? Dimension tables contain descriptive attributes that provide context for the facts. They answer the "who, what, when, where, why, and how" questions about business events.

Characteristics:
  • Descriptive Attributes: Names, descriptions, categories, hierarchies
  • Smaller Volume: Fewer records compared to fact tables
  • Wide Tables: Many columns with descriptive information
  • Surrogate Keys: System-generated unique identifiers

Star Schema

Star schema is the simplest dimensional modeling approach where a central fact table is directly connected to multiple dimension tables, creating a star-like structure.

Star Schema: Central fact table surrounded by dimension tables in a star pattern

Snowflake Schema

Snowflake schema extends the star schema by normalizing dimension tables into multiple related tables, creating a snowflake-like structure with branching hierarchies.

Snowflake Schema: Normalized dimension tables create hierarchical sub-dimensions

Slowly Changing Dimensions (SCD)

Managing Dimensional Changes Over Time: Slowly Changing Dimensions are categorical attributes that change over time, and managing that change is very hard to handle. Organizations must choose appropriate strategies based on business requirements and compliance needs.

SCD Type 0: No Change

  • Dimension table will not be changing
  • Used for static attributes like date of birth, social security numbers
  • Once loaded, data remains unchanged

SCD Type 1: Overwrite

  • Simply overwrite the old data with new data
  • No historical tracking
  • Uses UPSERT operations (Update + Insert)
  • It will do the update and insert operation

Data Lakes vs Data Warehouses

Are we replacing Data Warehouse with Data Lake?

Not exactly, but in a way we are improving it. In data warehouse, we can contain structured data but with data lake we can have semi-structured and unstructured data as well.

Data Lake serves as central storage for diverse data sources and use cases
Area Data Lake Data Warehouse
Data Types Structured, Semi-structured, and Unstructured data Only structured data with predefined schemas
Schema Approach Schema-on-Read: Define structure when accessing data Schema-on-Write: Define structure before storing data
Storage Cost Low cost, commodity storage, pay-as-you-go Higher cost, specialized storage infrastructure
Processing Speed Variable - depends on query optimization and data organization Fast - optimized for analytical queries with pre-processing
Data Quality Raw data quality - may require cleaning and validation High quality - data is cleaned and validated before storage
Flexibility High agility - can store any data format and evolve schema Lower flexibility - requires predefined structure and planning

Curious Question: Should we discard data warehouse then?

No! We will see why and how to use combination of Data Lake and Data Warehouse. This new approach is called Lakehouse.

Medallion Architecture

Note: We do apply medallion architecture in our cloud data engineering architecture. This pattern is widely adopted by organizations using modern data platforms like Databricks, Azure, and AWS.

A medallion architecture is a data design pattern used to logically organize data in a lakehouse, with the goal of incrementally and progressively improving the structure and quality of data as it flows through each layer of the architecture. It is also referred to as "multi-hop" architecture.

Bronze Layer (Raw Data)

Raw Data Integration

  • Landing zone for raw data from external source systems
  • No data cleanup or validation performed
  • Table structures mirror source systems "as-is"
  • Additional metadata columns: load date/time, process ID
  • Focus on Change Data Capture and historical archive
  • Data stored in native format (JSON, CSV, etc.)

Silver Layer (Cleaned and Transformed)

Filtered, Cleaned, Augmented

  • Data cleaning and validation performed
  • Matched, merged, conformed, and cleansed data
  • Enterprise view of key business entities and transactions
  • Enables self-service analytics and ad-hoc reporting
  • Source for data scientists and analysts
  • ELT methodology with "just-enough" transformations

Gold Layer (Business-Level Aggregates)

Business Intelligence Ready

  • Consumption-ready, project-specific databases
  • Denormalized, read-optimized data models
  • Final data transformations and quality rules applied
  • Subject areas: Customer Analytics, Product Quality, Inventory
  • Optimized for reporting and business intelligence

File Formats & Big Data Technologies

Row-based vs Column-based Formats

The fundamental difference in how data is organized and stored significantly impacts query performance, storage efficiency, and system optimization:

Aspect OLTP (Row-based Storage) OLAP (Column-based Storage)
File Formats CSV, AVRO, JSON - optimized for transactional updates Parquet, ORC, Columnar - optimized for analytical reads because column-based storage is good for read operations
Data Organization Data written row by row: [ID=1, Name="John", Age=25, Salary=50000] then [ID=2, Name="Jane", Age=30, Salary=60000] Data written column by column: [ID: 1,2,3...] [Name: "John","Jane","Bob"...] [Age: 25,30,28...] [Salary: 50000,60000,55000...]
Storage Pattern All attributes of one record stored together physically on disk Same attributes from all records stored together physically on disk
Query Performance Excellent for single record retrieval because all data for one row is stored together Superior for aggregations because you only read needed columns, not entire rows
Compression Efficiency Limited compression because mixed data types within rows High compression ratios because similar data types in columns compress better
Write Operations Fast INSERT/UPDATE because entire row written at once Slower INSERT/UPDATE because data scattered across multiple column files
Read Operations Read entire row even if you only need few columns Read only required columns, reducing I/O significantly
Use Case Optimization Transaction processing, real-time updates, operational systems Analytics, reporting, data warehousing, business intelligence because column-based is good for OLAP

Understanding Delta File Concept

In Delta file, we have a transaction log that enables powerful capabilities:

  • Versioning: Because we can do versioning, I can travel back to my previous version if needed
  • Schema Evolution: It allows me to change/evolve the schema easily without breaking existing queries
  • ACID Transactions: Ensures data consistency and reliability for all operations
  • Time Travel: Access any historical state of data for debugging and analysis

Benefits of Delta File

  • Transaction Log Benefits: The entire metadata is stored in transaction log, providing a single source of truth for all data operations and enabling complete audit trails
  • Schema Flexibility: It allows me to change/evolve the schema easily. Add, remove, or modify columns without rewriting entire datasets
  • ACID Compliance: It is handy in terms of ACID - ensuring atomicity, consistency, isolation, and durability for all data operations
  • Performance Optimization: It is very efficient and has everything from traditional databases. Query is also fast because of Parquet file, the only addition is transaction log
  • Data Reliability: Automatic handling of failures, concurrent writes, and data validation

Delta format combines the best of both worlds: the performance of Parquet files for analytics with the reliability and features of traditional databases through its transaction log.

Big Data Framework Ecosystem

We have different frameworks based on the type of data we work with:

1. Apache Kafka

Distributed Event Streaming Platform: Apache Kafka is a distributed event store and stream-processing platform designed to provide a unified, high-throughput, low-latency platform for handling real-time data feeds. Work with streaming data and real-time big data processing for building real-time data pipelines and streaming applications.

2. Apache Airflow

Workflow Orchestration Platform: Apache Airflow is an open-source workflow management platform for data engineering pipelines that allows programmatic authoring and scheduling of workflows. Used for orchestration when we need to build so many pipelines with complex logic that needs to be run in specific order.

3. Apache Hive

Data Warehouse System: Apache Hive is a distributed, fault-tolerant data warehouse system that enables analytics at a massive scale and facilitates reading, writing, and managing petabytes of data using SQL. Create external tables and provides SQL-like interface to query large datasets stored in distributed systems.

4. Apache Spark

Unified Analytics Engine: Apache Spark is a multi-language engine for executing data engineering, data science, and machine learning on single-node machines or clusters. Hero product and powerful distributed computing framework used extensively in Databricks for data processing and analytics.

Let's see how distributed computing with Spark works?

We ask multiple machines to work together for complete computation. It becomes a cluster with improved efficiency and speed.

Apache Spark Cluster Architecture: Master Node coordinates with Cluster Manager to distribute tasks to Worker Nodes

How Spark Architecture Works

This is Spark Architecture. The human will tell the Driver node (inside Master Node). Now, the driver node will orchestrate and tell the worker node to do certain task, similarly the other worker nodes will be notified of their task. This is called the cluster of machines where bunch of machines will together work to do certain task and brain is driver node which is responsible for orchestrating the work. This is called Spark architecture.

Detailed Spark Architecture Components

  • Master Node: Contains the Driver Program that manages the overall Spark application and coordinates all operations
  • Driver Program: The main program that creates SparkContext and defines transformations and actions on RDDs
  • Spark Context: The entry point for all Spark functionality, coordinates with cluster manager to acquire resources
  • Cluster Manager: External service responsible for acquiring resources and managing the cluster (can be Spark's own, YARN, Mesos, or Kubernetes)
  • Worker Nodes: Machines in the cluster that execute the actual data processing tasks
  • Task Executors: Processes on worker nodes that run individual tasks and cache data
  • Cache Storage: In-memory storage on each worker node for fast data access and reuse

Benefits of Spark Distributed Architecture

  • Fault Tolerance: If one worker node fails, tasks are automatically reassigned to other nodes
  • Horizontal Scalability: Add more worker nodes to handle larger datasets and increase processing power
  • In-Memory Processing: Cache frequently used data in memory across the cluster for faster processing
  • Resource Optimization: Cluster manager efficiently allocates resources based on workload requirements
  • Parallel Processing: Multiple tasks execute simultaneously across different nodes for improved performance

Cloud Data Engineering

On-Premise vs Cloud Infrastructure

Organizations must choose between managing their own data infrastructure or leveraging cloud services. Each approach has distinct advantages and use cases.

Aspect On-Premise Cloud
Infrastructure Ownership Organization owns and maintains physical hardware, servers, networking equipment Cloud provider owns and maintains all physical infrastructure
Capital Investment High upfront costs for hardware, software licenses, and data center setup Pay-as-you-go model with no upfront infrastructure investment
Scalability Limited by physical hardware capacity, requires planning for peak loads Virtually unlimited scalability, can scale up or down based on demand
Control & Security Complete control over security policies, data location, and compliance Shared responsibility model, relies on cloud provider's security measures
Maintenance Organization responsible for updates, patches, hardware replacement Cloud provider handles maintenance, updates, and infrastructure management
Data Location Data stays within organization's physical premises Data stored in cloud provider's data centers globally

When to Use On-Premise:

  • Sensitive Data: Highly confidential or regulated data that must stay within organization boundaries
  • Compliance Requirements: Strict regulatory requirements that mandate on-premise data storage
  • Predictable Workloads: Consistent resource usage patterns with minimal variation
  • Complete Control: Need for full control over hardware, software, and security configurations

When to Use Cloud:

  • Variable Workloads: Fluctuating resource demands that benefit from elastic scaling
  • Rapid Deployment: Need to quickly provision resources and deploy applications
  • Cost Optimization: Want to avoid large capital expenditures and pay only for resources used
  • Global Accessibility: Require access to data and applications from multiple geographic locations

Current Industry Trend: 90% of data is stored in cloud, 10% sensitive data kept on-premises. This hybrid approach allows organizations to leverage cloud benefits while maintaining control over critical data.

What is Cloud Computing?

Cloud computing is the delivery of computing services including servers, storage, databases, networking, software, analytics, and intelligence over the Internet ("the cloud") to offer faster innovation, flexible resources, and economies of scale. As an organization, demand varies so we can use as many resources as needed depending upon the workload from cloud platforms like Azure, AWS and GCP.

Cloud Computing Benefits:

  • Cost Efficiency: Eliminate capital expenditure on hardware and software, pay only for resources consumed
  • Scalability: Instantly scale computing resources up or down based on demand without infrastructure planning
  • Global Accessibility: Access applications and data from anywhere with internet connectivity
  • Automatic Updates: Cloud providers handle software updates, security patches, and infrastructure maintenance
  • Disaster Recovery: Built-in backup and recovery capabilities with geographic redundancy
  • Innovation Speed: Rapid deployment of new services and technologies without hardware procurement delays
  • Resource Optimization: Efficient resource utilization through shared infrastructure and automated management
  • Reduced IT Burden: Minimize internal IT infrastructure management and focus on core business objectives
  • Enhanced Security: Enterprise-grade security measures and compliance certifications from cloud providers
  • Environmental Impact: Reduced carbon footprint through shared, optimized data center resources

Note: Cloud Storage is very cheap because of data lakes! Everything like storage and compute operates from cloud servers.

Understanding the Azure Data Pipeline Architecture

This Microsoft-recommended architecture demonstrates a complete modern analytics platform built on Azure Databricks at its core, seamlessly integrated with Azure services for data ingestion, processing, storage, and serving.

Data Flow Stages

Data Flow Stages

1. Data Sources and Ingestion

The architecture begins with diverse data sources feeding into the platform:

Azure Databricks serves as the central processing engine with multiple capabilities:

MLflow: Machine learning lifecycle management for tracking experiments, models, and deployments.

3. Storage Layer - Medallion Architecture

Azure Data Lake Storage Gen2 implements the medallion architecture with Delta Lake format:

Bronze Layer

Raw data ingestion zone storing data in its original format. Maintains full fidelity and history of source data.

Silver Layer

Cleaned and validated data with standardized formats. Data is deduplicated, validated, and enriched for enterprise consumption.

Gold Layer

Business-ready aggregates optimized for specific use cases. Contains curated datasets for analytics and reporting.

4. Serving Layer

Processed data serves multiple consumption patterns:

Azure Databricks SQL Warehouses: SQL analytics with Photon-powered Delta Engine for high-performance queries.

Power BI: Business intelligence and visualization with Direct Lake mode for optimal performance through Fabric mirroring.

Azure Machine Learning: Model deployment and serving for real-time predictions.

Azure Kubernetes Service (AKS): Container-based deployment for scalable ML model serving.
